# R12.1 — AI-summarized digests + NL query over local state

**Parent strategy:** [../../PROGRAM-REVIEW-2026-04.md](../../PROGRAM-REVIEW-2026-04.md)
**Tier:** B
**Task ID:** `f6d96087-5f66-4344-ae5a-68f679bb5168`

## 1. Goal

Let the user ask *"what changed on project X this week?"* or *"where am I
overspending?"* and get a grounded answer drawn from the local SQLite
state. Plus scheduled digests (daily/weekly) summarizing each project.
Realizes the "intelligence-first" commitment from the program review.

## 3. Architecture

The LLM sees **structured tools**, not raw SQL. Reuses the MCP tool
schemas as the function-calling interface to avoid a parallel schema.

- User supplies a key via settings (Anthropic API, OpenAI, or a local
  OpenAI-compatible endpoint like Ollama).
- The server forwards the prompt + MCP tool schemas; the model calls
  tools; the server executes them against the local DB; the model
  composes the final answer.
- All calls are local-first: no DB content leaves the machine except
  to the provider the user configured.

## 4. Data model changes

Migration in `schema.ts`:

- New table `llm_configs` (id TEXT PK, provider ENUM('anthropic',
  'openai','ollama','openai_compatible'), model TEXT, base_url TEXT NULL,
  api_key_enc TEXT, active INT, created_at TEXT). Encrypted at rest
  using a key derived from a user-supplied passphrase or OS keychain
  (implementation note — see §10).
- New table `digests` (id TEXT PK, scope_type ENUM('project','fleet'),
  scope_id TEXT NULL, period ENUM('daily','weekly'), generated_at TEXT,
  content_md TEXT, cost_usd REAL, tokens_in INT, tokens_out INT).
- New table `nl_queries` (id TEXT PK, query TEXT, answer_md TEXT,
  tool_calls_json TEXT, cost_usd REAL, created_at TEXT) — transient
  history; pruned after N days.

## 5. Server surface

New module `server/llm/`:
- `provider.ts` — tiny abstraction over Anthropic + OpenAI-compatible
  chat completions + tool calls. No framework (no LangChain). Reuse
  existing `anthropic` SDK if already a dep; otherwise raw fetch.
- `query.ts` — orchestrates: expand query → loop through tool calls → return answer.
- `digest.ts` — scheduled job; for each project, gather last-period
  facts via internal tool calls, produce markdown digest, store.
- `scheduler.ts` — cron-style loop (reuse the R11.2 scheduler if built).

Reuse MCP tool schemas as the function-calling interface (one source of
truth; a tool the LLM can call is a tool an agent can call).

REST in `routes.ts`:
- `POST /api/llm/query` — body: `{ query, scope? }` → streams answer.
- `GET /api/llm/digests?scope_type=&scope_id=`, `POST /api/llm/digests/run`.
- CRUD `/api/llm/configs`.

MCP tools:
- `ask_vibe_dash` — agents can ask natural-language questions too.
- `generate_digest` — trigger a digest run.

WebSocket: `digest_ready`, `llm_query_token` (for streaming).

## 9. Out-of-scope

- Fine-tuning, embeddings, vector search — not needed at this scope.
- Writing to the DB via NL ("mark X as done") — read-only for v1.
- Multi-turn conversations with memory beyond a single query.

## 10. Security / privacy

- API keys are encrypted at rest; never returned by list endpoints
  (masked).
- Explicit UI disclosure that the configured provider receives the DB
  contents (task titles, activity text, etc.) during a query.
- Opt-in; not enabled by default.
- Honor CLAUDE.md rule: never hardcode real keys in tests; use
  `"test-api-key-placeholder"`.
