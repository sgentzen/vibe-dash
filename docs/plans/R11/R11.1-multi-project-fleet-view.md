# R11.1 ‚Äî Multi-project fleet view

**Parent strategy:** [../../PROGRAM-REVIEW-2026-04.md](../../PROGRAM-REVIEW-2026-04.md)
**Tier:** A (highest leverage)
**Task ID:** `f8c1073d-09da-4941-8928-d1eead2723a2`

## 1. Goal

Ship a meta-dashboard that shows **all local projects at once** ‚Äî health,
active agents, burn, blockers, cost ‚Äî so the user can answer *"what's the
status of everything I'm working on?"* without switching project context.
This is the primary-intent feature of the April 2026 program review.

## 3. Data model changes

Add migration in [server/db/schema.ts](../../../server/db/schema.ts):

- New table `discovered_projects` (id TEXT PK, path TEXT UNIQUE, detected_at
  TEXT, linked_project_id TEXT NULL FK‚Üíprojects.id, ignored INT DEFAULT 0) ‚Äî
  supports passive discovery when the user opts in.
- New view or cached rollup table `project_health_snapshot` (project_id PK,
  snapshot_at TEXT, active_agents INT, open_blockers INT, overdue_tasks INT,
  in_progress_tasks INT, cost_7d REAL, cost_30d REAL, milestone_progress_pct
  REAL) refreshed every N minutes by a scheduler. Materializing avoids N
  round-trips in the fleet view.

Migrations must follow the R9b numbered-migration pattern already in
`schema.ts` ‚Äî append, never mutate prior migrations.

## 4. Server surface

- New module `server/db/fleet.ts`:
  - `getFleetOverview(db): FleetOverview[]` ‚Äî joins projects +
    project_health_snapshot + recent activity digest.
  - `refreshFleetSnapshots(db)` ‚Äî computes the rollup, called by scheduler
    and on mutation WebSocket events (debounced).
  - `discoverProjects(db, roots: string[])` ‚Äî walks configured roots,
    detects `.git` + `.mcp.json` + `CLAUDE.md`, inserts into
    `discovered_projects`.
- New REST endpoints in [server/routes.ts](../../../server/routes.ts):
  - `GET /api/fleet` ‚Äî fleet overview (uses `statsLimiter`).
  - `GET /api/fleet/discovery` ‚Äî list discovered-but-unlinked projects.
  - `POST /api/fleet/discovery/:id/link` ‚Äî link to existing project OR
    create new project from discovery.
  - `POST /api/fleet/discovery/:id/ignore`.
  - `POST /api/fleet/refresh` ‚Äî force snapshot refresh.
- New MCP tools in [server/mcp/tools.ts](../../../server/mcp/tools.ts):
  - `get_fleet_overview` ‚Äî so agents can self-check fleet health.
  - `discover_projects` ‚Äî agent-initiated discovery trigger.
- WebSocket events: `fleet_snapshot_refreshed`,
  `fleet_project_discovered`.

## 5. Client surface

- New component `src/components/FleetView.tsx` ‚Äî grid of project cards,
  each showing name, health chip (on-track / at-risk / behind), active
  agents, open blockers, 7d cost, in-progress task count, sparkline of
  recent activity. Click ‚Üí drills into that project's existing views.
- Register in [src/App.tsx](../../../src/App.tsx) as the **default** root
  view when more than one project exists. Existing single-project views
  become the drill-down.
- New `src/components/DiscoveryBanner.tsx` ‚Äî shows discovered-but-unlinked
  projects with Link / Ignore actions.
- Store changes in [src/store.tsx](../../../src/store.tsx): add
  `fleet` slice with `overview`, `discoveries`, `loading`.

Design principle: information density, not polish. Sparklines, tight
typography, keyboard-navigable.

- Cost-deep-dive (handled by R11.2).

## 9. Deprecation companion

`ExecutiveView` today tries to be a global view but only covers one
project at a time. Once FleetView lands, either:
(a) retire ExecutiveView entirely, folding its status chips into
FleetView cards; or
(b) demote ExecutiveView to a single-project detail tab.
Decide during R11.4. Do not delete yet.
