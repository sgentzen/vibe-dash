### Remove (1 table, 2 route files)

| Feature | Classification | Rationale |
|---------|---------------|-----------|
| `saved_filters` table | **Remove** | No frontend component reads `GET /api/filters`. No hook in `useApi.ts` calls the endpoint. DB functions (`createSavedFilter`, `listSavedFilters`, `deleteSavedFilter`) are exported but called only from `server/routes/filters.ts`. The feature was never surfaced to users. URL-parameter filtering in `search_tasks` covers the use case adequately. |
| `server/routes/sprints.ts` | **Remove** | Registered nowhere â€” unreachable dead code. Migration `004_sprints_to_milestones` migrated sprint data; the sprint concept is fully superseded by milestones. |

### Deprecate â†’ Remove in R12 (1 table/feature)

| Feature | Classification | Rationale |
|---------|---------------|-----------|
| `alert_rules` | **Deprecate** | Evaluated at only 3 event types (task_completed, blocker_reported, comment_added) while webhooks cover all events. No frontend UI to manage rules â€” agents/users can't author them from the dashboard. The notification generation it drives is consumed by the frontend, but webhooks + a custom notification endpoint would cover this more cleanly. Keep for one release to allow migration; remove in R12 after providing a migration path. |

However, it creates tasks (not just activity/cost events) and has a distinct use case (incident-to-task automation). Keep for now, evaluate for merge with ingestion in R12.
