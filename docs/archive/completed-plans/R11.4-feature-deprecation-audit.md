## 1. Goal

Actively **remove** features that don't defend themselves against the
April 2026 strategic bet: *"PM layer for AI agents, multi-project status
first."* Produce an audit report, reach alignment with Scott on what
survives, then execute removals with clean migration paths.

## 3. Audit criteria (apply to every feature)

For each MCP tool, REST endpoint, DB table, and dashboard view, answer:

1. **Primary-intent fit** — does this directly serve the multi-project
   status view, or a clearly supporting capability?
2. **Usage** — has it actually been called? Check `activity` rows, logs,
   and `mcp-tools.test.ts` coverage as proxies.
3. **Overlap** — is there another feature doing the same job? (Webhooks
   vs alert_rules; saved_filters vs search_tasks params; etc.)
4. **Maintenance cost** — lines of code, test weight, schema surface.
5. **Replacement after R11.1–R11.3** — once FleetView + Cost v2 +
   Ingestion land, does this feature become redundant?

Apply the criteria; classify each as **Keep / Merge / Deprecate / Remove**.

## 4. Pre-identified candidates (from program review)

Not pre-judged — each must still go through §3:

- `suggest_agent` MCP tool + routing logic — low signal, adds surface.
- `report_working_on` / file_locks — cool demo, unclear usage.
- `alert_rules` table — overlaps with `webhooks`.
- `saved_filters` — overlaps with URL-param filtering.
- Multiple overlapping cost endpoints post-R11.2
  (`get_cost_timeseries` vs scorecards).
- `templates` + `create_project_from_template` — has it earned its
  weight vs a plain example project?
- `ExecutiveView` — subsumed by FleetView (R11.1).
- OpenAPI/Swagger surface — keep only if R11.3's `/api/ingest` docs
  depend on it.
- Worktree integration — keep (ties directly to multi-agent PM), but
  audit cost of schema.

## 5. Deliverables

1. **Audit report** — `docs/plans/R11/R11.4-audit-report.md` with every
   tool/table/view scored against §3 and classified.
2. **Decision doc** — `docs/plans/R11/R11.4-decisions.md` after Scott
   sign-off: what survives, what's deprecated-with-deadline, what's
   removed immediately.
3. **Removal PRs** — one per logical grouping so diffs stay reviewable:
   - Remove dead code paths (src + server + db).
   - Drop unused tables (with a *down* migration in `schema.ts`).
   - Update `tests/`, `CLAUDE.md`, `README.md`.
4. **Migration notes** — `docs/MIGRATION-R11.md` for any user-visible
   removals (MCP tools, REST endpoints).

## 6. Execution order

1. Land R11.1, R11.2, R11.3 first so the deprecation picture is real,
   not speculative.
2. Produce audit report.
3. Scott signs off on the classification.
4. Ship removal PRs.

## 7. Test plan

- Every removal must delete its tests; `npm test` must still pass with
  reduced suite.
- Integration test: full fresh-install smoke test (create project →
  log task → see in FleetView) must still pass.
- No new test files introduced by this task; only deletions + updates.

## 8. Verification

- `npm test` green on reduced surface.
- `npm run build` green.
- `grep` for removed symbols returns zero (no dangling refs).
- MIGRATION doc lists every breaking change.

## 9. Out-of-scope

- Adding new features (belongs to other R11/R12/R13 tasks).
- Performance optimization of kept features.
- Renames without removal (cosmetic churn).

## 10. Non-obvious risks

- **Tests passing after deletion is not proof the feature was unused.**
  Some features may be depended on by external users' workflows we don't
  see. Cross-check with GitHub issues, CHANGELOG references, and the
  MCP-SETUP guide before removing anything user-facing. Prefer a
  deprecation warning + next-release removal over hard breaks.
- Schema removals are the most dangerous; if in doubt, mark the table
  `deprecated_*` and stop writing to it for one release before dropping.
